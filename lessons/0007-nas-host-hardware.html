NAS host hardware: board, CPU, RAM, controllers

Disks and pool layout are chosen. The box around them still matters: motherboard, CPU, RAM, how disks attach, and a separate boot drive. This lesson is the shopping checklist for that host — not the OS or the *arr apps (those get their own lessons).

~13 min Concept Win: know the five host decisions that protect a home ZFS NAS

Why OS/software is separate

Hardware answers “can ZFS see healthy disks and enough RAM?” Software answers “TrueNAS vs plain Linux, where do Jellyfin and Sonarr run?” Mixing them overloads working memory. Finish the host checklist here; next storage lesson will pick the NAS OS and where apps live.

1. RAM first (then ECC)

TrueNAS’s documented minimum is 8 GB; that boots a lab box, not a comfortable home NAS with media apps. The Hardware Guide’s rule of thumb: at least 8 GB for basic use with up to eight drives, then more as you add disks, clients, apps, or VMs. For your mission (files + later Jellyfin/*arr on or beside the NAS), treat 16 GB as the practical floor and 32 GB as the comfortable target.

ECC is strongly recommended: interference can flip a bit in RAM; without ECC that bad bit can be written and checksummed as if it were truth. Official nuance (worth remembering): many home systems run non-ECC successfully, but ECC is another integrity layer — and it does constrain which CPU/board pairs you can buy (Hardware Guide — ECC). If budget forces non-ECC, do not skip the 3-2-1 backups from Lesson 4.

Board shopping tip

Want ECC? Verify the pair: CPU model and chipset/manual must both list ECC / ECC UDIMM support. A “server looking” board without matching ECC CPU support is just expensive non-ECC.

2. How disks connect (motherboard vs HBA)

Small builds (roughly your 6–8 bay RAIDZ2): onboard SATA is fine if you have enough 6 Gbps ports. Many boards mix slow 3 Gbps and fast 6 Gbps ports — use the fast ones for data disks (Hardware Guide — controllers).

Need more ports than the board has? Add a Broadcom/LSI-class HBA flashed to IT / Target mode so each disk is passed through. Community classics: LSI 9207/9300-class cards (often cheap used). If a card only speaks hardware RAID, avoid it or force true JBOD/passthrough — ZFS already is your RAID (ZFS Primer).

Hardware RAID is the wrong friend

RAID cards can hide SMART data, serials, and real disk health — or lie with a write cache. For TrueNAS, passthrough beats “RAID-5 on the card.”

3. Boot drive ≠ data pool

Install the OS on its own SSD (official minimum ~20 GB; bigger leaves room for boot environments). Do not carve the boot volume out of your RAIDZ2 data disks. Avoid USB sticks / flaky SATA DOMs for the boot pool — TrueNAS writes enough that cheap USB media often dies early (Hardware Guide — boot).

4. CPU and motherboard (what to optimise for)

You are not buying a gaming rig. Priorities for this mission:

  • Enough cores for background work — scrubs, compression, later apps. A mid-range modern x86_64 is plenty for file serving.
  • AES-NI (almost everything modern has it) if you turn on dataset encryption later.
  • SATA count + one free PCIe slot for an HBA if you outgrow onboard ports.
  • Quiet, 24/7-friendly board/case/PSU — Gold-rated PSU sized for spin-up of all disks; UPS is wise (graceful shutdown on power loss).
  • Optional: IPMI/BMC on server boards for remote power/console — nice, not required for a desk-side home NAS.

4a. Jellyfin on this box? Decide the GPU now

Earlier drafts of this course “parked” transcoding until the media lesson. That was wrong for shopping: once the CPU/board is bought, you either have a usable video engine or you don’t. What can wait is Jellyfin’s dashboard settings and Docker device passthrough — not the silicon.

Two different jobs get confused:

  • Direct play — the client plays the file as-is. Almost no CPU/GPU work on the NAS. Ideal when TVs/phones understand your codecs.
  • Transcode — the server re-encodes on the fly (wrong codec, bitrate limit, subtitle burn-in, remote stream). Without a GPU this melts a CPU and wakes the whole house; with one it stays quiet (Jellyfin — Hardware Acceleration).

If Jellyfin (or equivalent) will share this host and you expect any transcoding, pick a platform with a supported accelerator before you buy:

PathWhat to buyTrade-off

Course default when media shares the NAS

Prefer a modern Intel CPU with an enabled iGPU (Quick Sync / VA-API path). It usually beats “buy a loud discrete GPU for one TV” for a living-room-adjacent box. Still optimise for silence: an iGPU does not excuse a screaming workstation cooler. If media will live on a separate compute host later, the NAS CPU can stay boring — but decide that split before purchasing, not after.

Still parked for the media-stack lesson: how many streams you need, HDR tone-map quirks, and exact Jellyfin/Docker flags. Those tune the choice; they don’t erase the need to have some supported GPU in the cart.

5. Network and USB (quick)

1 GbE is enough to start; 2.5 GbE is a cheap upgrade if LAN and clients can use it. Do not put the primary ZFS pool on USB disks — serial-number weirdness and disconnects make USB a backup cable, not a pool cable (Hardware Guide).

Host checklist (print-friendly)

  1. Bay count matches Lesson 5 layout (e.g. 6–8 for RAIDZ2).
  2. RAM: ≥16 GB (32 GB if apps share the host); ECC if the pair allows.
  3. Enough 6 Gbps SATA or IT-mode HBA — no hardware RAID arrays.
  4. Separate SSD for boot (≥20 GB).
  5. CMR SATA data disks from Lesson 6.
  6. Decent PSU + consider a UPS.
  7. If Jellyfin shares this box: iGPU/dGPU path chosen (or explicit “direct play only”).
  8. OS install details: next lesson — but the CPU/GPU SKU is decided here.
