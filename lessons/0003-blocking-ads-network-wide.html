Blocking ads & tracking network-wide

One setting, every device: phones, smart TVs, that chatty IoT plug — all with no software installed on any of them. The trick is to block at the layer they all quietly rely on: DNS. This lesson explains how DNS ad-blocking works, then turns on AdGuard Home on your Flint 2.

~15 min Concept + build Win: ad-blocking for the whole house at the DNS layer

Recap: DNS is the phone book every device uses

Before your phone can load example.com, it must turn that name into an IP address. It asks a resolver — and it only ever asks one, whichever your DHCP server handed it when it joined the network. That single chokepoint is the whole idea: if we control the resolver, we control what every device is allowed to look up.

How blocking at DNS works

An ad or tracker isn't magic — it's just a request to a domain like ads.example.com or track.metrics.io. A DNS sinkhole keeps a big blocklist of those domains. When a device asks for one, the resolver simply refuses — returns "no such address" — so the connection is never even attempted. The ad never downloads; the tracker never phones home.

Because every device on the network already funnels its lookups through this one resolver, a single blocklist covers the entire house — no per-device app, no fiddling with a TV that won't let you install anything.

What DNS blocking can and can't do

Great at: killing whole ad/tracker domains for every device at once — especially TVs and IoT you can't put an extension on.

Can't do: block an ad served from the same domain as the content (e.g. YouTube pre-rolls), or hide page elements the way a browser extension (uBlock Origin) can. DNS blocking and a browser blocker are complementary, not rivals — run both.

AdGuard is a bundled application on the Flint 2, so there's no extra hardware and — importantly — no second DHCP server. Your Flint 2 stays the one DHCP server; it just starts advertising itself (192.168.8.1) as the DNS resolver, and AdGuard answers on the router. One network, one DHCP, one resolver — clean.

Encrypted DNS: the piece that quietly matters most

Plain DNS is clear text — your ISP can read and log every single domain every device looks up, and many monetise exactly that. AdGuard forwards the queries it doesn't block to an upstream over DoH/DoT, so the ISP sees an encrypted blob, not your browsing. Blocking still works because it happens on your resolver first, before anything leaves the house. Sovereignty, not just fewer ads.

The build

1. Turn on AdGuard Home

On the Flint 2: APPLICATIONS → AdGuard Home. Enable it. GL.iNet wires it in as the router's resolver automatically. Set an admin password when prompted, then open its own dashboard (the app links to it).

2. Point it at an encrypted upstream

In AdGuard's Settings → DNS settings → Upstream DNS servers, use an encrypted upstream instead of plain 53. Good choices:

https://dns.quad9.net/dns-query      # Quad9 (blocks known-malicious too)
https://cloudflare-dns.com/dns-query # Cloudflare 1.1.1.1

Click Test upstreams to confirm they answer, then Apply. Leave "load balancing" on so it spreads queries across them.

3. Add blocklists

Filters → DNS blocklists. AdGuard ships with its default list; add the community-standard OISD list for broad coverage. Resist stacking ten aggressive lists on day one — overlap adds little and over-blocking breaks sites, which is annoying to debug. Start lean, add later if ads slip through.

4. Confirm every device actually uses it

DHCP hands out the new DNS setting only when a device renews its lease. Reboot a device (or toggle its Wi-Fi) so it picks up 192.168.8.1 as DNS, then load a known ad-heavy site. Watch AdGuard's Query Log — you should see queries arriving, some tagged blocked. A dashboard that stays empty means devices are still using their old DNS.

The bypass gotcha — some devices ignore your resolver

DNS blocking only works if devices actually ask AdGuard. Two common escapees:

Hardcoded DNS: some gear (Google/Android, many smart TVs) ships with 8.8.8.8 baked in and ignores DHCP. Fix on the Flint 2 with a firewall rule that redirects all outbound port-53 traffic back to the router — so even hardcoded queries get caught.

Browser DoH: Chrome/Firefox may run their own encrypted DNS ("Secure DNS"), sailing right past AdGuard. Turn "Secure DNS" / "DNS-over-HTTPS" off in the browser so it uses the system resolver.

Reference card with the exact rule: Force all DNS to AdGuard (Flint 2 / LuCI).

The payoff: ad-blocking anywhere

Remember the full-tunnel choice from Lesson 2? Here's where it pays off. On a full tunnel, your phone's AllowedIPs is 0.0.0.0/0, so all its traffic — including DNS — routes home through 10.1.0.1 to the Flint 2, where AdGuard is waiting. Turn on WireGuard on cellular and you get the same ad-blocking on the train that you get on the sofa. That's the full stack clicking together: one resolver, reachable from anywhere, over a tunnel only you can open.
