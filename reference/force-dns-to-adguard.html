Force all DNS to AdGuard

A DNAT "DNS hijack" that catches devices with hardcoded DNS (8.8.8.8 and friends) and drags their port-53 queries back to the Flint 2, where AdGuard is listening. Built for quick reference — skim it, print it, return to it.

Flint 2 · LAN 192.168.8.0/24 Resolver: 192.168.8.1 Layer: firewall (DNAT)

What it does (one line)

Any LAN packet aimed at any DNS server on port 53 is rewritten to hit 192.168.8.1:53. Conntrack rewrites the reply's source back to the original address, so the device never notices — hardcoded DNS just silently starts obeying AdGuard.

Get into LuCI

GL.iNet admin panel → System → Advanced Settings, or go straight to http://192.168.8.1/cgi-bin/luci. Log in as root with your router admin password.

Option A — LuCI GUI

Network → Firewall → Port Forwards → Add. A port-forward is a DNAT rule. Set:

FieldValue
NameIntercept-DNS
Restrict to address familyIPv4 only (see IPv6 caveat)
ProtocolTCP, UDP
Source zonelan
External port53
Destination zonelan
Internal IP address192.168.8.1
Internal port53

Save, then Save & Apply.

Option B — SSH (copy-paste, most reliable)

ssh root@192.168.8.1, then:

uci set firewall.dns_int=redirect
uci set firewall.dns_int.name='Intercept-DNS'
uci set firewall.dns_int.src='lan'
uci set firewall.dns_int.src_dport='53'
uci set firewall.dns_int.proto='tcp udp'
uci set firewall.dns_int.target='DNAT'
uci set firewall.dns_int.dest='lan'
uci set firewall.dns_int.dest_ip='192.168.8.1'
uci set firewall.dns_int.dest_port='53'
uci commit firewall
/etc/init.d/firewall restart

Equivalent block in /etc/config/firewall:

config redirect
	option name 'Intercept-DNS'
	option src 'lan'
	option src_dport '53'
	option proto 'tcp udp'
	option target 'DNAT'
	option dest 'lan'
	option dest_ip '192.168.8.1'
	option dest_port '53'

Test

From a LAN device, aim a query at a public resolver on purpose:

nslookup example.com 8.8.8.8

It still answers — but the answer now comes from AdGuard. Confirm the lookup appears in AdGuard's Query Log. Before the rule it would never show up there. An empty log = the redirect isn't matching.

Caveats & sibling rules

IPv6 is a separate door

If the LAN hands out IPv6, a device can bypass via an IPv6 DNS server (e.g. 2001:4860:4860::8888). Add a second redirect scoped to IPv6 pointing at the router's LAN IPv6 address. Heads-up: OpenWrt bug #18387 — with rx-gro-list on (24.10 default) IPv6 DNAT can leak packets; workaround ethtool -K <lan-if> rx-gro-list off. If you don't rely on IPv6, the IPv4 rule covers the common offenders.

DoT — block port 853

DNS-over-TLS skips port 53 entirely. Add a firewall reject rule for lan → wan on TCP/UDP 853. Clients then fall back to plain 53, which this redirect catches.

DoH — can't catch at the firewall

DNS-over-HTTPS (port 443) is indistinguishable from normal web traffic. Handle it by turning Secure DNS / DNS-over-HTTPS off in browsers, not at the firewall.

No loop with AdGuard

The rule is scoped to src 'lan'. AdGuard's own upstream queries originate from the router, not the lan zone, so they're not redirected. Keep the scope as-is.
