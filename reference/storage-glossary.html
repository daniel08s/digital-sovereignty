Storage Glossary

Canonical definitions for the storage & backup lessons. Lessons use these terms exactly as defined here. Print-friendly; skim and return.

Redundancy
Extra disks (or copies inside one system) so the system can keep serving data when a drive dies. Survives hardware failure at that box — not fire, ransomware, or "I deleted the wrong folder."
Backup
A separate, recoverable copy of data, ideally independent of the live system. Survives mistakes, malware, and disasters that wipe the primary. A mirror of two disks in one NAS is redundancy, not a backup.
3-2-1 rule
Keep 3 copies of important data, on 2 different kinds of media, with 1 copy off-site. Baseline recommended by CISA. Primary NAS + external drive + cloud/offsite = classic home shape.
RAID
Combining multiple disks into one logical volume for capacity and/or redundancy. Traditional RAID protects against a dead drive; it does not detect silent corruption, and it is not a backup.
Bit rot
Silent data corruption: a drive is still "healthy," but bits flipped (age, cosmic rays, firmware bugs). Without checksums you may not notice until you open a file years later.
ZFS
A combined filesystem + volume manager with end-to-end checksums. On a redundant pool it can self-heal corrupt blocks. Primary home platform for this course: TrueNAS (OpenZFS).
Checksum
A fingerprint of a data block. ZFS stores one with every block and verifies it on every read. Mismatch = corruption detected.
Self-healing
When a checksum fails and a good copy still exists (mirror / RAIDZ), ZFS rewrites the bad block from the good one — automatically.
Scrub
A scheduled full read of the pool to verify every checksum. Catches rarely-read corrupt blocks early. TrueNAS schedules monthly scrubs by default.
Pool (zpool)
The top-level ZFS storage container. Built from one or more vdevs. Lose a whole vdev → lose the pool.
vdev
A group of disks that provides one stripe of capacity/redundancy (mirror, RAIDZ1/2/3, or stripe). Topology is chosen at create time and is hard to change later — plan before you buy.
RAIDZ1 / RAIDZ2 / RAIDZ3
ZFS parity layouts. The number is how many disks you can lose per vdev without losing data. Home sweet spot is often RAIDZ2 (survive two failures). RAIDZ1 is discouraged for large modern drives — long rebuild windows.
Mirror
Two (or more) disks holding identical data. Great random IOPS; costs 50% capacity on a two-way mirror. Prefer for VMs/databases; RAIDZ2 often better for bulk media.
Snapshot
A cheap, instant point-in-time view of a dataset (thanks to copy-on-write). Great for "oops, deleted that" recovery on the same pool — still not an off-site backup until you replicate it elsewhere.
Replication
Sending ZFS snapshots to another pool (another machine or off-site). Turns local history into a real second/third copy.
Resilver
Rebuilding a replaced disk by copying only used blocks from remaining redundancy. Faster than classic RAID rebuilds, but still a danger window — especially on large drives.
